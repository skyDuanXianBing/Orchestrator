# 需求概述

本需求旨在“阶段详情弹窗（Phase Detail Modal）”内，实时展示任意子代理（sub-agent）在某个阶段内发生的操作流（类似 tool-call trace），包括但不限于：Read / Glob / Grep / Command / Edited / Artifact 等。该操作流以“时间线（timeline）”形式按顺序追加展示，并与现有阶段汇总字段（`changed_files` / `commands_executed` / `artifact_pointers` 等）并存：

- 时间线：用于可观测性与排障，记录“每一次操作”的关键元数据，按阶段保存最近 500 条。
- 汇总字段：用于阶段结果概览，仍保持“去重 + 累积”的现有行为，不被时间线替代。

非目标：

- 不在时间线中展示文件内容、命令输出全文、或任何可能泄露敏感信息的原始数据。
- 不改变既有阶段状态、耗时、变更文件、命令列表、制品指针等汇总字段的语义，仅补齐缺失的操作记录来源。

### 概念与数据流

1) 子代理执行任意工具/操作时，产出“操作时间线条目（Operation Timeline Entry）”。
2) 服务端将条目追加到对应 `phaseId` 的时间线中，并执行截断（仅保留最近 500 条）。
3) 服务端通过 SSE 推送增量事件（或通过 `BLACKBOARD_UPDATED` 驱动刷新），前端在阶段详情弹窗内实时渲染。

### 操作分类（Taxonomy）

操作类型（`op_type`）采用稳定枚举字符串，便于前端展示与筛选：

| op_type | 含义 | 示例 label | 典型 target.kind | 典型 meta |
|---|---|---|---|---|
| `read.file` | 读取文件（只记录元信息） | `Read file` | `path` | `line_count`/`byte_count`/`truncated` |
| `read.dir` | 读取目录列表（只记录元信息） | `List directory` | `path` | `entry_count`/`truncated` |
| `search.glob` | Glob 匹配文件 | `Glob files` | `pattern` | `match_count`/`sample_paths[]` |
| `search.grep` | Grep/rg 搜索内容（不记录命中行） | `Search pattern` | `pattern` | `match_count`/`file_count` |
| `exec.command` | 执行命令（bash/tool） | `Run command` | `command` | `exit_code`/`duration_ms`/`cwd` |
| `fs.edit` | 编辑/写入文件（apply_patch 等） | `Edit file` | `path` | `changed_paths[]`/`diff_stats` |
| `artifact.add` | 产出制品/指针（doc、报告、构建产物等） | `Add artifact` | `artifact` | `pointer`/`kind` |
| `net.fetch` | 网络获取（webfetch/tavily 等） | `Fetch URL` | `url` | `status_code`/`bytes`/`duration_ms` |
| `ai.context.query` | 文档/上下文查询（Context7 等） | `Query docs` | `id` | `source`/`hit_count` |
| `phase.note` | 阶段内部的说明性事件（可选） | `Note` | `id` | `text_length` |
| `unknown` | 兜底类型（不可识别时） | `Operation` | `id` | `raw_type` |

说明：

- `exec.command` 仅记录“已脱敏”的命令摘要与元数据；禁止保存 stdout/stderr 全文。
- `read.file` / `search.grep` 禁止保存原始文本内容；仅允许保存计数、文件路径样本等。

### 时间线条目载荷（Payload Schema）

时间线条目为“追加型事件”，建议具备可去重与可排序能力：

```json
{
  "op_id": "phase_3:000123",
  "seq": 123,
  "ts": "2026-02-24T17:27:12Z",
  "phase_id": "phase_3",
  "agent": "impl_green_coder",
  "op_type": "exec.command",
  "state": "COMPLETED",
  "label": "Run command",
  "target": {
    "kind": "command",
    "value": "pnpm test",
    "display": "pnpm test"
  },
  "meta": {
    "duration_ms": 1432,
    "exit_code": 0,
    "cwd": "packages/server",
    "truncated": false
  },
  "redaction": {
    "applied": true,
    "rules": ["command.secret_flags", "env.assignments"],
    "replaced_count": 1
  }
}
```

字段约束：

- `op_id`：同一 `phase_id` 内全局唯一；服务端生成，客户端用于幂等去重。
- `seq`：同一 `phase_id` 内单调递增整数；用于严格排序（优先级高于 `ts`）。
- `ts`：ISO 时间戳字符串。
- `state`：`STARTED` / `COMPLETED` / `FAILED`。若仅记录一次也可直接为 `COMPLETED`。
- `label`：短文本（面向 UI）；必须为脱敏后的可展示字符串。
- `target`：可选，描述操作对象；`value` 为机器值（脱敏后），`display` 为展示值（可进一步裁剪）。
- `meta`：可选，操作结果元信息；必须满足数据安全约束（见“数据安全约束”）。

### 与现有 PhaseRecord 汇总字段的映射

现有 `PhaseRecord`（`packages/shared/src/types/blackboard.ts`）汇总字段保持语义不变，时间线作为补充：

- `changed_files`：从 `fs.edit`（或更细粒度的 `file.edited`）条目中抽取 `target.kind=path` 或 `meta.changed_paths[]`，以“去重集合”累积。
- `commands_executed`：从 `exec.command` 条目抽取 `target.display`（脱敏后），以“去重集合”累积。
- `artifact_pointers`：从 `artifact.add` 条目抽取 `meta.pointer` 或 `target.value`，以“去重集合”累积。
- `summary` / `error_summary`：不从时间线推导；仍由阶段结束/失败时的聚合产出。

建议：时间线与汇总字段采用同一套脱敏函数，以避免“汇总可见但时间线被遮蔽/反之”的不一致。

## 用户故事

1) 作为使用者，我希望在“阶段详情”里实时看到该阶段正在发生的每一次子代理操作（Read/Glob/Grep/Command/Edited/Artifact 等），以便理解进度与定位卡住的位置。

2) 作为开发/排障人员，我希望操作流能稳定排序、可去重、可截断，并在 SSE 断开时仍能通过轮询回放到最新状态，以便复现与排查问题。

3) 作为安全负责人，我希望时间线展示不会泄露密钥、token、密码、cookie、私有文件内容或命令输出，从而避免把敏感信息暴露在 UI 与持久化数据中。

## 功能列表 (Must Have / Should Have)

Must Have

- 为每个 `phaseId` 新增“操作时间线”概念：仅保留最近 500 条条目，按 `seq` 顺序追加。
- 覆盖“任意子代理操作”的记录：至少包含 Read / Glob / Grep / Command / Edited / Artifact 六类，并能扩展更多类型。
- 定义统一的时间线条目 schema（含 `op_id`、`seq`、`ts`、`agent`、`phase_id`、`op_type`、`label`、可选 `target` 与 `meta`）。
- 实时性：当阶段详情弹窗打开时，新条目能实时出现在 UI 中（不需要手动刷新页面）。
- 汇总字段不变：`changed_files` / `commands_executed` / `artifact_pointers` 继续去重累积；时间线不替代汇总。
- 数据安全：对命令行与可疑字段进行脱敏；禁止记录文件内容与命令输出全文。

Should Have

- 条目支持 `state=STARTED/COMPLETED/FAILED`，便于展示长耗时操作的“进行中”。
- UI 支持筛选/搜索（按 `op_type`、关键字、失败状态），并支持一键复制“脱敏后的命令摘要”。
- 当发生截断（>500）时，UI 显示“已仅保留最近 500 条”的提示（可选包含丢弃计数）。
- 将时间线与汇总字段的“脱敏规则版本”写入 `redaction.rules[]`，便于审计与回归测试。

## 验收标准 (Checklist)

- [ ] 操作分类与 schema 在文档中定义清晰：包含字段含义、必填/可选、排序与幂等等约束。
- [ ] 任意子代理操作均可映射为时间线条目（至少覆盖 Read/Glob/Grep/Command/Edited/Artifact），且能扩展。
- [ ] 实时性：当服务端产生新条目时，前端阶段详情弹窗在 P95 ≤ 2 秒内展示（含网络抖动）；典型情况下 ≤ 1 秒。
- [ ] 顺序：同一 `phaseId` 内按 `seq` 递增展示；在乱序到达时，前端能按 `seq` 重排且不闪烁。
- [ ] 去重：若同一条目重复投递（SSE 重连/重放），客户端基于 `op_id` 幂等去重；汇总字段继续基于“脱敏后的字符串值”去重累积。
- [ ] 截断：同一 `phaseId` 时间线超过 500 条时，服务端丢弃最旧条目，仅保留最新 500 条；不会影响汇总字段的完整性。
- [ ] 展示：阶段详情弹窗同时展示“汇总字段区（现有）+ 操作时间线区（新增）”；时间线条目至少显示 `ts`、`op_type`、`label`、`agent`，并可展开查看 `target/meta`。
- [ ] 数据安全：命令行、URL、header、环境变量等敏感信息被脱敏；时间线不包含文件内容、stdout/stderr 全文、或任何未经脱敏的 secret。

### 数据安全约束（脱敏与禁止项）

禁止写入时间线（无论 UI 是否展示）：

- 文件原始内容（Read 结果正文、Grep 命中行正文）。
- 命令输出全文（stdout/stderr）、以及包含敏感信息的日志片段。
- HTTP 请求/响应的敏感头（`Authorization`、`Cookie`、`Set-Cookie` 等）原值。

必须脱敏的常见来源（写入 `target.value/display` 与 `meta` 前执行）：

- 命令行参数：`--token`/`--password`/`--pass`/`--secret`/`--key`/`--api-key`/`--access-key`/`--header Authorization:...` 等，将其值替换为 `[REDACTED]`。
- 环境变量赋值：形如 `PASSWORD=...`、`TOKEN=...`、`AWS_SECRET_ACCESS_KEY=...`，保留键名，值替换为 `[REDACTED]`。
- URL 查询参数：`?token=...&key=...&sig=...` 等，将值替换为 `[REDACTED]`。
- 类似密钥的长随机串：满足“长度阈值 + 字符集特征”的字符串（例如 Base64/Hex 形式），在不影响可读性的前提下进行掩码（如保留前 4 后 4）。

允许写入的安全元数据示例：

- `duration_ms`、`exit_code`、`match_count`、`file_count`、`byte_count`、`line_count`、`truncated`、`cwd`（如为绝对路径可选改为相对/归一化）。
