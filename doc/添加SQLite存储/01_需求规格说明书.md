# 需求概述
本需求用于将 Orchestrator 后端在运行过程中产生的数据（任务状态、阶段状态、SSE 事件/日志、黑板快照）持久化到 SQLite 数据库中，以支持：任务列表/详情查询、故障排查审计、SSE 事件回放、进程重启后的状态恢复（可选）。

本次仅定义数据模型与表结构、数据写入与读取的数据流（不包含任何实现代码）。

### 数据范围
- 任务级数据：`taskId/category/mode/userRequirement/projectPath/docPath/pipelineStatus/currentPhaseIndex/createdAt/updatedAt`
- 阶段级数据：每个 `phaseId` 的运行时状态（`status/retryCount/startedAt/finishedAt/sessionId/...`）及其产物（changed files、commands、artifacts）
- 事件/日志：后端向前端推送的 SSE 事件（`SSEEventType`）以及 `LOG_MESSAGE`
- 黑板快照：每次状态更新后的 `BlackboardJson`（用于审计与快速回放/恢复）

### 设计原则
- 以 `task_id` 为主键贯穿所有表，保证可按任务维度聚合查询。
- SSE 事件必须可按严格顺序回放（同一任务内有单调递增的序列号）。
- 在 FAST 模式下优先简单可落地：允许将数组/对象以 JSON 文本方式存储；提供 Should Have 的“可查询性增强”规范。

### 逻辑实体与关系
- `tasks` 1 — N `phases`
- `tasks` 1 — N `sse_events`
- `tasks` 1 — N `blackboard_snapshots`（可选；也可只存 latest）

## 表结构（SQLite Schema 规格）

### 1) tasks（任务表）
用途：承载任务的“当前状态”（用于列表、详情、恢复），对应 `TaskSummary` + `PipelineState` 的关键字段。

字段：
- `task_id` TEXT PRIMARY KEY
- `category` TEXT NOT NULL  （取值：A/B/C/D，对应 `Category`）
- `mode` TEXT NOT NULL      （取值：FAST/BALANCED/COMPREHENSIVE/HARDENING，对应 `Mode`）
- `user_requirement` TEXT NOT NULL
- `project_path` TEXT NOT NULL
- `doc_path` TEXT NOT NULL
- `pipeline_status` TEXT NOT NULL  （取值：IDLE/RUNNING/PAUSED_FOR_REVIEW/COMPLETED/FAILED/ABORTED，对应 `PipelineStatus`）
- `current_phase_index` INTEGER NOT NULL
- `current_phase_id` TEXT NULL     （冗余字段，便于列表展示）
- `created_at` TEXT NOT NULL        （ISO-8601）
- `updated_at` TEXT NOT NULL        （ISO-8601）

索引建议：
- `idx_tasks_updated_at(updated_at)`
- `idx_tasks_pipeline_status(pipeline_status)`

### 2) phases（阶段表）
用途：承载每个任务的每个阶段的“当前状态”，对应 `PhaseRuntime`。

字段：
- `id` INTEGER PRIMARY KEY AUTOINCREMENT
- `task_id` TEXT NOT NULL  REFERENCES tasks(task_id) ON DELETE CASCADE
- `phase_id` TEXT NOT NULL
- `phase_type` TEXT NOT NULL      （AGENT_GATE/HUMAN_REVIEW，对应 `PhaseType`）
- `agent` TEXT NULL               （子代理名，对应 `AgentName`；HUMAN_REVIEW 为 NULL）
- `status` TEXT NOT NULL          （PENDING/IN_PROGRESS/SUCCESS/FAILED/SKIPPED/APPROVED_BY_HUMAN，对应 `PhaseStatus`）
- `retry_count` INTEGER NOT NULL DEFAULT 0
- `started_at` TEXT NULL          （ISO-8601）
- `finished_at` TEXT NULL         （ISO-8601）
- `session_id` TEXT NULL          （opencode session id）
- `opencode_task_id` TEXT NULL    （runner 内部绑定的 task id，如有）
- `error_summary` TEXT NULL
- `summary` TEXT NULL
- `artifacts_json` TEXT NOT NULL DEFAULT '[]'         （JSON 数组）
- `changed_files_json` TEXT NOT NULL DEFAULT '[]'     （JSON 数组）
- `commands_executed_json` TEXT NOT NULL DEFAULT '[]' （JSON 数组）

约束：
- UNIQUE(`task_id`, `phase_id`)：同一任务同一阶段只保留一行（更新覆盖）。

索引建议：
- `idx_phases_task(task_id)`
- `idx_phases_task_status(task_id, status)`

### 3) sse_events（SSE 事件表/日志表）
用途：持久化事件流，支持任务详情页重放、排障审计、断线重连补发。

字段：
- `id` INTEGER PRIMARY KEY AUTOINCREMENT
- `task_id` TEXT NOT NULL REFERENCES tasks(task_id) ON DELETE CASCADE
- `seq` INTEGER NOT NULL            （同一 task 单调递增序列号，用于严格排序与断点续传）
- `type` TEXT NOT NULL              （对应 `SSEEventType`）
- `timestamp` TEXT NOT NULL         （ISO-8601）
- `phase_id` TEXT NULL              （若事件与阶段相关，写入 phaseId；否则 NULL）
- `level` TEXT NULL                 （仅当 type=log_message 时使用：info/warn/error/debug）
- `agent` TEXT NULL                 （仅当 type=log_message 或阶段事件时使用）
- `message` TEXT NULL               （仅当 type=log_message 时使用）
- `data_json` TEXT NOT NULL         （事件 data 的 JSON，原样存储）

约束：
- UNIQUE(`task_id`, `seq`)：保证任务内事件序列唯一。

索引建议：
- `idx_events_task_seq(task_id, seq)`
- `idx_events_task_time(task_id, timestamp)`
- `idx_events_type(type)`

### 4) blackboard_snapshots（黑板快照表，Should Have）
用途：保留黑板历史版本用于审计与回滚；FAST 模式可先不启用，仅存 latest（见下一节）。

字段：
- `id` INTEGER PRIMARY KEY AUTOINCREMENT
- `task_id` TEXT NOT NULL REFERENCES tasks(task_id) ON DELETE CASCADE
- `version` INTEGER NOT NULL        （从 1 递增）
- `reason` TEXT NOT NULL            （如：initialize/updatePhase/advance/human_review/agent_result）
- `blackboard_json` TEXT NOT NULL   （`BlackboardJson` 原样 JSON）
- `created_at` TEXT NOT NULL        （ISO-8601）

约束：
- UNIQUE(`task_id`, `version`)

索引建议：
- `idx_snapshots_task_version(task_id, version)`

### 5) tasks.blackboard_latest_json（替代方案，Must Have）
若暂不做快照表，则在 `tasks` 增加以下字段以满足“详情查询与恢复”：
- `blackboard_latest_json` TEXT NOT NULL DEFAULT '{}' （`BlackboardJson` latest）
- `blackboard_version` INTEGER NOT NULL DEFAULT 0

说明：该方案写入更简单，但无法保留历史版本。

## 数据流（写入/读取）

### 写入时机
1) 创建任务（TaskManager.createTask）
- 写入 `tasks`（pipeline_status=IDLE，current_phase_index=0，current_phase_id='phase_0'）
- 初始化写入 `phases`：为该任务的所有 phaseId 插入行（或按需惰性插入），状态为 PENDING
- 写入 `blackboard_latest_json`（或插入 `blackboard_snapshots` version=1）
- 追加 `sse_events`：可选记录 `pipeline_created`（若不新增事件类型则仅记录 LOG_MESSAGE）

2) 状态机更新（updatePhase / advance / setCurrentPhaseIndex）
- 更新 `tasks.updated_at/current_phase_index/current_phase_id`（如可计算）
- 更新对应 `phases` 行（status、时间戳、summary、error_summary、json 数组列）
- 更新 `tasks.blackboard_latest_json` 与 `blackboard_version+=1`（或插入 `blackboard_snapshots`）
- 追加 `sse_events`：若当前逻辑已 emit `BLACKBOARD_UPDATED`，则同步落库。

3) EventBus/SSE emit
- 每次 emit SSEEvent 时，都追加写入 `sse_events`
- 对于 `LOG_MESSAGE`：将 `level/agent/phase_id/message` 落到专用列，同时 `data_json` 原样保留

4) 删除任务（TaskManager.deleteTask）
- 删除 `tasks` 对应行（级联删除 `phases/sse_events/snapshots`）

### 读取场景
1) 任务列表（对应 `TaskListResponse`）
- 从 `tasks` 查询基础字段；`completedPhases` 可通过 `phases` 聚合（status in SUCCESS/APPROVED_BY_HUMAN）得到

2) 任务详情（对应 `TaskDetailResponse`）
- 从 `tasks` 获取基础字段 + `blackboard_latest_json`（或取快照表 latest）
- 可选：同时从 `phases` 取出各阶段状态用于 UI 展示（减少 JSON 解析依赖）

3) SSE 事件回放/断点续传
- 客户端携带 `lastSeq`，服务端从 `sse_events` 读取 `seq > lastSeq` 的事件按序返回

## 兼容性与约束
- 时间字段统一使用 ISO-8601 字符串（与现有类型保持一致）。
- JSON 字段必须存储“可被原样 JSON.parse”的文本。
- 外键约束需开启（SQLite `PRAGMA foreign_keys = ON`），并要求级联删除生效。

## 用户故事
1) 作为后端开发者，我希望任务/阶段/事件有持久化记录，以便排查流水线失败原因并可重启恢复。
2) 作为前端用户，我希望刷新页面后仍能看到历史任务与阶段进度，并能回放关键日志与事件。
3) 作为运维/审计人员，我希望可以按 taskId 查询完整执行轨迹（阶段状态变化 + 事件流 + 黑板内容）。

## 功能列表 (Must Have / Should Have)

Must Have:
- 定义并落地 SQLite 表：`tasks`、`phases`、`sse_events`
- `tasks` 能支撑任务列表与任务详情（包含 blackboard latest 或等价信息）
- `sse_events` 支撑按 `task_id + seq` 严格顺序回放
- 基础索引与唯一约束：`tasks.task_id`、`phases(task_id, phase_id)`、`sse_events(task_id, seq)`

Should Have:
- `blackboard_snapshots` 保存黑板历史版本（支持审计/回滚）
- 将 `artifacts/changed_files/commands` 进一步规范化为子表（提高可查询性）
- 任务删除支持软删除（保留审计记录）与保留策略（按天数/数量清理）
- 提供断线重连的 `lastSeq` 补发策略与接口规格（若未来需要）

## 验收标准 (Checklist)
- [ ] SQLite 中存在并可创建以下表：`tasks`、`phases`、`sse_events`（以及可选 `blackboard_snapshots`）
- [ ] `tasks` 一行可完整表示 `TaskSummary` 所需字段，并可通过 `task_id` 获取详情
- [ ] `phases` 可通过 `(task_id, phase_id)` 唯一定位并更新，且能表达 `PhaseRuntime` 的所有字段（含数组 JSON）
- [ ] `sse_events` 支持同一 `task_id` 下按 `seq` 严格递增排序回放；`(task_id, seq)` 唯一
- [ ] 事件数据 `data_json` 为可解析 JSON，且包含与现有 `SSEEvent.data` 等价的信息
- [ ] 开启外键后，删除任务会级联删除其 phases/events/snapshots（如启用）
