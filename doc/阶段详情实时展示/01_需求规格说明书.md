# 需求概述

本需求针对 Web 端“阶段详情（Phase Detail）弹窗”在流水线执行过程中需要**实时**展示阶段执行信息的问题，要求在阶段运行时无需用户手动刷新即可持续更新关键字段，并在阶段结束后冻结为最终结果。

## 目标与范围

- 范围：仅增强现有阶段详情弹窗（PhaseDetailModal）在**任务执行中**的展示与刷新机制；已完成任务仍按最终数据展示。
- 数据来源：以任务 Blackboard 中 `PhaseRecord` 为单一事实来源（字段包括：`status`, `started_at`, `finished_at`, `changed_files`, `commands_executed`, `artifact_pointers`, `summary`, `error_summary`）。

## “实时”的产品定义（需要明确的口径）

1) 更新触发方式（优先级）：

- Must：SSE 推送驱动（与现有任务事件流保持一致）。当后端检测到阶段记录（PhaseRecord）发生变化时，通过 SSE 发送更新事件，前端据此刷新弹窗内容。
- Should：在 SSE 不可用/断线重连失败时，提供降级轮询（Polling）同步最终一致性。

2) 可接受延迟（端到端）：

- Must：阶段状态/列表/摘要等变更在后端落盘后，前端弹窗**1 秒内**可见（P95）。
- Must：在网络抖动/重连场景下，允许短暂延迟但不应超过 **3 秒**（P99），并最终与服务端状态一致。

3) 是否展示“增量/进行中”的列表：

- Must：当 `commands_executed` / `changed_files` / `artifact_pointers` 在阶段执行中逐步产生时，弹窗允许展示**部分结果**并持续追加更新。
- Must：列表展示应保证“可读性优先”，允许去重（同一条重复上报时不重复展示），并保持稳定排序（按产生时间/追加顺序）。

4) `finished_at` 为 null 时耗时行为：

- Must：若 `started_at` 已存在且 `finished_at == null` 且阶段处于运行态（如 `IN_PROGRESS`），弹窗耗时显示为“动态计时”，以客户端本地定时器每 1 秒更新一次：`now - started_at`。
- Must：当 `finished_at` 写入后，耗时冻结为：`finished_at - started_at`。
- Must：若 `started_at` 为空，耗时显示为未知（例如 `--`），不进行计时。

## 非目标（Non-goals）

- 不新增/替换现有阶段详情弹窗的整体布局与交互范式（仅补齐实时刷新与字段展示的正确性）。
- 不要求在本需求中实现“命令实时日志流”或“文件 diff 预览”等重功能（可作为后续需求）。
- 不要求跨任务/跨页面的全局实时看板能力（仅限当前任务、当前弹窗）。

## 关键名词

- 阶段（Phase）：流水线中的一个步骤，对应 `phase_0/phase_1/...`。
- PhaseRecord：Blackboard 中每个阶段的记录对象。
- 实时：由 SSE 推送或轮询触发的数据刷新 + 进行中耗时的动态计时。

# 用户故事

1) 作为使用者，我在任务运行时打开某个阶段的详情弹窗，希望看到阶段状态、耗时以及执行产物（命令/变更文件/工件指针）会自动更新，不需要我手动刷新页面。

2) 作为使用者，我希望阶段结束后弹窗显示最终结果并保持稳定（不再跳动），方便我复盘。

3) 作为开发/运维，我希望前后端对“什么时候推送更新、推送什么内容”有明确约定，避免 UI 频繁抖动或漏更新。

# 功能列表 (Must Have / Should Have)

## Must Have

- 阶段详情弹窗在阶段执行中可实时更新：`status`、`summary`、`error_summary`、`commands_executed`、`changed_files`、`artifact_pointers`。
- 实时更新触发：后端通过 SSE 推送阶段记录更新（可为 BLACKBOARD 级别事件或专用 Phase 级别事件），前端接收后更新弹窗内容。
- 进行中耗时：当 `finished_at == null` 且 `started_at` 存在时，弹窗每 1 秒更新耗时；阶段完成时耗时冻结。
- 增量列表展示：执行中逐步追加的列表在弹窗可见；同一条数据重复上报时应避免重复展示（允许前端做去重）。
- 一致性：断线重连后，弹窗最终应与服务端当前 Blackboard 中 PhaseRecord 一致（允许重连后拉取一次全量以对齐）。

## Should Have

- SSE 降级轮询：当 SSE 不可用时，按固定间隔轮询拉取最新阶段数据（建议 2 秒），恢复 SSE 后停止轮询。
- 节流/合并渲染：短时间内多次更新（例如 200ms 内多条）可合并渲染，避免弹窗抖动。
- “最后更新时间”提示（用于可观测性）：弹窗显示最近一次数据更新时间戳。

# 验收标准 (Checklist)

- [ ] 当任务正在执行且某阶段状态从 `PENDING` → `IN_PROGRESS`，打开该阶段详情弹窗后，状态在 1 秒内更新为 `IN_PROGRESS`。
- [ ] 当 `finished_at == null` 且 `started_at` 不为空时，弹窗耗时每 1 秒递增；当 `finished_at` 写入后，耗时冻结且不再递增。
- [ ] 当后端在阶段执行中逐步追加 `commands_executed` 时，弹窗列表在 1 秒内出现新增条目；重复条目不会无限重复展示。
- [ ] 当后端在阶段执行中逐步追加 `changed_files` 或 `artifact_pointers` 时，弹窗在 1 秒内展示新增条目；阶段完成后列表不回退、不闪烁。
- [ ] 当网络短暂中断并恢复（模拟断线/重连），弹窗在 3 秒内恢复更新，并最终与服务端当前 PhaseRecord 一致。
- [ ] 当阶段失败（`status == FAILED`），弹窗在 1 秒内展示失败状态与 `error_summary`（若存在）。

## 后端行为验收（对齐前后端契约）

- [ ] 后端在阶段执行中对 Blackboard/PhaseRecord 的写入是**增量可见**的：字段变化（状态、列表追加、摘要更新）会在执行期间陆续落盘，而不是只在结束时一次性写入。
- [ ] SSE 至少具备一种可用的更新事件：
  - 方案 A（可接受）：`BLACKBOARD_UPDATED` 事件，包含任务标识与最新 Blackboard（或足够重建 PhaseRecord 的数据）。
  - 方案 B（推荐）：专用 `PHASE_RECORD_UPDATED` 事件，仅携带 `{ task_id, phase_id, phase_record }` 或 diff。
- [ ] SSE 事件顺序或版本控制具备“后到覆盖前到”的确定性（例如 `updated_at`/`revision` 字段或服务端按写入顺序发送），避免前端回放旧数据导致回退。
