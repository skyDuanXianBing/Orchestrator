# 需求概述
本需求针对 Web 前端（Vue 3 + Vite，`packages/web`）的「Create New Task」页面进行 UI 改造：将当前以纯文本输入的「项目路径（可选）」字段，替换为“可输入 + 可选择”的项目路径选择器。选择器通过弹出式下拉/弹窗提供可选路径（例如最近使用），同时保留手动输入作为兜底。

约束与前提：
1. 运行环境为浏览器（非 Electron/桌面应用），不得调用本机原生文件系统选择对话框，也不得在前端直接枚举本机目录。
2. 所有用户可见文案必须使用 `vue-i18n` 提供（禁止硬编码）。
3. 如需从后端获取路径列表，必须通过前端现有的集中式 HTTP 封装调用（禁止在组件内直接 `fetch/axios`）。

目标：
1. 降低手动输入路径的成本与错误率。
2. 提供最近使用路径的快速选择能力。
3. 不影响现有创建任务的表单提交与后端接口契约（字段仍为字符串，可为空）。

非目标（本期不做）：
1. 在浏览器端扫描/列出用户文件系统目录结构。
2. 提供 OS 级文件夹选择器。
3. 修改后端任务创建接口的字段命名或必填规则。

## 用户故事
1. 作为创建任务的用户，我希望能从最近用过的项目路径里快速选择，避免重复输入。
2. 作为创建任务的用户，当列表中没有我想要的路径时，我仍然可以手动输入一个路径。
3. 作为创建任务的用户，我希望选择器交互清晰可控（能打开/关闭、能清除选择、不会遮挡表单关键区域）。

## 功能列表 (Must Have / Should Have)
Must Have：
1. 将「项目路径（可选）」改为“输入框 + 触发按钮”的组合控件；触发按钮打开弹出层（popover/dropdown 或 modal 均可）。
2. 弹出层展示“可选路径列表”，默认来源为“最近使用路径”（存储于浏览器 localStorage）。
3. 支持从列表点击选择：选中后将路径写入输入框，并关闭弹出层。
4. 支持手动输入：用户可直接在输入框键入/粘贴路径；该字段依然为可选（允许为空）。
5. 与现有创建任务流程兼容：提交时仍以字符串形式提交项目路径；不因选择器改造引入必填校验。
6. i18n：组件/页面新增或变更的所有文案均通过 `vue-i18n` key 提供。

Should Have：
1. 路径列表支持搜索/过滤（对“最近使用路径”进行前端过滤）。
2. 支持“清除当前路径”操作（一键清空输入框）。
3. 近期路径管理：最大保留条数（例如 10 条）、去重、按最近使用时间排序。
4. 可扩展的数据源：在不破坏 Must Have 行为的前提下，支持切换/补充为后端 API 提供的路径候选（例如工作区/服务器可访问的项目目录）。
5. 可访问性：键盘可操作（打开/关闭/上下选择/回车确认/ESC 关闭），并具备合理的 aria 属性。

## 验收标准 (Checklist)
以下每条均需可通过手工验收或 E2E 测试验证：
1. 「Create New Task」页面中不再出现仅有的纯文本“项目路径”输入；替换为“输入框 + 选择按钮”。
2. 点击选择按钮会打开弹出层；再次点击按钮或点击弹出层外部区域会关闭弹出层。
3. 弹出层打开时，能看到“最近使用路径”列表：
   - 当 localStorage 中存在记录时，按“最近使用优先”顺序展示。
   - 当不存在记录时，展示空态（空态文案来自 i18n）。
4. 点击任意一条路径后：
   - 输入框值更新为该路径；
   - 弹出层关闭；
   - 表单其余字段不被重置。
5. 用户可在输入框手动输入/粘贴任意字符串（包含空格与常见路径分隔符），且不会被组件强制改写格式。
6. 该字段保持“可选”：输入框为空时也允许提交创建任务（与改造前行为一致）。
7. 当创建任务成功（以前端判定成功响应为准）后：
   - 若“项目路径”非空，则该路径会被写入 localStorage 的“最近使用路径”集合；
   - 若为空，则不写入。
8. localStorage 最近路径集合满足：
   - 去重（同一路径不会出现多条）；
   - 有上限（例如 10 条，达到上限时淘汰最久未使用者）；
   - 再次使用某路径会被提升到列表首位。
9. 若浏览器禁用/不可用 localStorage（或写入失败）：
   - 选择器仍可手动输入并提交；
   - 组件不会导致页面崩溃；
   - 可选地提示非阻塞错误（提示文案来自 i18n）。
10. 前端代码层面：新增/改动的用户可见文本不出现硬编码字符串（均通过 `vue-i18n` key）。
11. 若实现了后端候选路径（Should Have #4）：
   - 请求必须通过集中式 HTTP 客户端；
   - 后端请求失败时不阻断手动输入与提交，并展示可恢复的错误状态（文案 i18n）。

补充说明：验收时可通过清理浏览器站点数据来验证“空列表/有列表/去重/上限淘汰”等场景。

# UI 行为规格
1. 组件形态
   - 左侧为可编辑输入框（展示当前项目路径字符串）。
   - 右侧为触发按钮（图标按钮或带文字按钮均可，但文案必须 i18n）。
2. 弹出层内容
   - 标题/分组（可选）：最近使用。
   - 列表项：显示完整路径字符串；超长时需可视截断但可通过悬浮/复制等方式看到完整值（如实现，文案 i18n）。
   - 空态：提示当前无可选路径，并引导用户手动输入（文案 i18n）。
3. 交互与状态
   - 打开：点击触发按钮；（Should Have）可支持输入框聚焦后按特定快捷键打开。
   - 关闭：点击外部区域、再次点击触发按钮、按 ESC。
   - 选择：点击列表项；（Should Have）支持键盘上下选择并回车确认。
   - 清除：（Should Have）提供清空按钮/操作，将输入框置空。
4. 表单集成
   - 该控件对外输出值为字符串（或空字符串/undefined，需与当前表单模型保持一致）。
   - 不新增强制校验；提交行为与改造前保持一致。

# 范围边界
In Scope：
1. 前端 UI 改造：将字段替换为选择器交互。
2. 最近路径的本地存储与读取（localStorage）。
3. i18n key 的补充与替换（确保无硬编码文案）。

Out of Scope：
1. 从用户本机文件系统读取目录/文件列表。
2. 任何原生文件夹选择对话框。
3. 服务端对目录的扫描/权限管理（除非后续单独立项）。

---

# 边界与异常场景
1. 路径包含特殊字符（空格、中文、`#`、`?` 等）：应原样保存与回填，不做 URL 编码/解码。
2. 超长路径显示：列表项需避免撑破布局；仍可选中并正确写回输入框。
3. 重复路径写入：再次使用同一路径应提升其“最近”顺序，而不是追加重复项。
4. localStorage 失败：读取/写入异常不应影响页面渲染与提交；仅降级为“无最近路径 + 仅手动输入”。
5. 多标签页并发：不同标签页写入最近路径后，刷新或重新打开弹出层应能读取到最新值（若未实现实时同步，至少刷新后生效）。
6. 安全性：最近路径为纯文本展示，必须避免将其当作 HTML 注入渲染（仅按文本渲染）。
